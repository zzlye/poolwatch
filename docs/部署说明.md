# 号池监控部署说明

本文说明如何把号池监控部署到 Linux 服务器，并通过域名和 HTTPS 供电脑、安卓手机访问。

## 一、准备工作

服务器需要满足以下条件：

- 已安装 Docker Engine 与 Docker Compose 插件。
- 域名的 A 或 AAAA 记录已指向服务器公网地址。
- 防火墙允许需要的端口。使用现有反向代理时通常只需保持原有 80、443 端口；使用随项目提供的 Caddy 时需开放 TCP 80、TCP 443 和 UDP 443。
- 服务器时间准确，建议启用系统时间同步。
- 具备 `openssl`，用于生成高强度随机密钥。

推荐把项目放在 `/opt/poolwatch`，把长期备份放在 `/srv/backups/poolwatch`。不要把真实 `.env`、数据库或备份提交到代码仓库。

## 二、生成部署配置

进入项目目录后运行：

```bash
cd /opt/poolwatch
sh scripts/generate-secrets.sh .env
chmod 600 .env
```

脚本发现 `.env` 已存在时会拒绝覆盖，防止误换正在使用的加密密钥。随后编辑 `.env`，至少修改 `PUBLIC_BASE_URL`；使用随项目提供的 Caddy 时还要修改 `DOMAIN` 和 `ACME_EMAIL`。

环境变量含义如下：

| 名称 | 是否必填 | 说明 |
| --- | --- | --- |
| `APP_ENCRYPTION_KEY` | 是 | Base64 编码的 32 字节密钥，用于保护渠道凭据、TOTP 和推送私钥。上线后必须保持不变并单独备份。 |
| `SETUP_TOKEN` | 是 | 创建唯一管理员时输入的初始化口令。管理员创建完成后应轮换为新的随机长串。 |
| `PUBLIC_BASE_URL` | 是 | 浏览器实际访问的完整 HTTPS 地址，例如 `https://monitor.example.com`，末尾不要加 `/`。 |
| `APP_PORT` | 否 | 映射到服务器回环地址的端口，默认 `8080`。 |
| `ALLOW_PRIVATE_TARGETS` | 否 | 默认 `false`。只有需要检测内网中的自有渠道时才设为 `true`。 |
| `TZ` | 否 | 时区，默认 `Asia/Shanghai`。 |
| `DOMAIN` | 使用随附 Caddy 时 | 对外域名，不包含协议和路径。 |
| `ACME_EMAIL` | 使用随附 Caddy 时 | 自动证书的到期和异常通知邮箱。 |

开启 `ALLOW_PRIVATE_TARGETS` 会允许管理员配置内网地址。只应在单管理员可信环境中开启，并通过防火墙限制应用可访问的内网范围；链路本地地址和已知云元数据、容器凭据地址仍会被阻止。

Web Push 不需要第三方 API Key。应用会自行生成 VAPID 密钥，并使用 `APP_ENCRYPTION_KEY` 加密后保存在业务数据卷中。迁移服务器时必须同时恢复数据卷和原加密主密钥，否则已有设备需要重新订阅通知。

Windows 本地开发可运行：

```powershell
powershell -ExecutionPolicy Bypass -File .\scripts\generate-secrets.ps1 -OutputPath .env
```

Windows 开发过程中产生的临时文件统一放在 `D:\tmp`；该路径不用于 Linux 正式服务器。

## 三、启动应用

使用已有反向代理时，先启动应用：

```bash
cd /opt/poolwatch
docker compose build --pull
docker compose up -d
docker compose ps
curl -fsS http://127.0.0.1:8080/healthz
```

`docker-compose.yml` 只把应用端口绑定到 `127.0.0.1`，公网不能绕过 HTTPS 直接访问。应用数据保存在名为 `poolwatch_app-data` 的数据卷中，容器更新和重建不会删除它。

首次通过域名打开应用时，输入 `.env` 中的 `SETUP_TOKEN`，然后创建管理员账号。创建成功后，可用下列命令生成新的高强度随机长串，把输出替换到 `.env` 的 `SETUP_TOKEN`：

```bash
openssl rand -hex 24
```

保存后重新创建应用容器：

```bash
docker compose up -d --force-recreate app
```

轮换后的初始化口令不会用于日常登录，也不能创建第二个管理员。保留非空随机值是为了满足部署配置校验；不要改成容易猜测的固定文字。

## 四、接入现有反向代理

### Nginx

项目提供了 `deploy/nginx.conf.example`。先把示例中的域名和证书路径替换为真实值，再放入 Nginx 配置目录。应用使用实时事件流，反向代理必须关闭响应缓冲并保留较长的读取超时；示例已经包含这些设置。

```bash
sudo nginx -t
sudo systemctl reload nginx
```

若修改了 `APP_PORT`，需要同步修改示例中的 `proxy_pass` 端口。

### 已有 Caddy

在现有 Caddyfile 中增加：

```caddyfile
monitor.example.com {
	reverse_proxy 127.0.0.1:8080 {
		flush_interval -1
	}
}
```

保存后验证并重载现有 Caddy。域名和端口应与 `.env` 的 `PUBLIC_BASE_URL`、`APP_PORT` 保持一致。

无论使用哪种反向代理，都应传递原始 `Host` 和 HTTPS 协议信息，不要缓存 `/api/` 响应或实时事件流。

### 反向代理也运行在容器中

容器内的 `127.0.0.1` 指向反向代理容器自身，不能用它访问号池监控。可把现有反向代理加入 `poolwatch_default` 网络，并把上游设为 `http://app:8080`：

```bash
docker network connect poolwatch_default 现有反向代理容器名
```

长期运行时应在现有反向代理自己的 Compose 配置中声明外部网络，避免重建容器后丢失连接：

```yaml
services:
  proxy:
    networks:
      - poolwatch

networks:
  poolwatch:
    external: true
    name: poolwatch_default
```

连接后，Nginx 使用 `proxy_pass http://app:8080;`，Caddy 使用 `reverse_proxy app:8080`。不要为了容器间访问而把应用端口公开到公网。

## 五、使用随项目提供的 Caddy

服务器没有现成反向代理时，可使用附带的组合配置：

```bash
cd /opt/poolwatch
docker compose -f docker-compose.yml -f deploy/docker-compose.caddy.yml up -d --build
docker compose -f docker-compose.yml -f deploy/docker-compose.caddy.yml ps
```

Caddy 会根据 `DOMAIN` 自动申请并续期证书。证书状态保存在 `poolwatch_caddy-data` 和 `poolwatch_caddy-config` 数据卷中。申请证书前确认：

- 域名已解析到当前服务器。
- 公网能够访问 80 和 443 端口。
- 服务器上没有其他程序占用 80 或 443 端口。
- `PUBLIC_BASE_URL` 为同一域名的 `https://` 地址。

查看证书申请或代理日志：

```bash
docker compose -f docker-compose.yml -f deploy/docker-compose.caddy.yml logs --tail=200 caddy
```

## 六、健康检查与运行维护

容器每 30 秒访问一次 `/healthz`。以下命令可检查状态和近期日志：

```bash
docker compose ps
curl -fsS http://127.0.0.1:8080/healthz
docker compose logs --tail=200 app
```

更新前先按照《备份与恢复》完成备份，然后执行：

```bash
docker compose build --pull
docker compose up -d
docker compose ps
```

更新不会自动删除数据卷。不要使用 `docker compose down -v`，其中的 `-v` 会删除业务数据。

## 七、通知与安装验收

Web Push 和安装到桌面或主屏幕都要求有效 HTTPS。部署完成后分别在 Windows Edge 或 Chrome、安卓 Chrome 中进行以下验收：

1. 打开域名并登录。
2. 在“推送设备”允许通知并发送测试通知。
3. 把应用安装到 Windows 桌面或安卓主屏幕。
4. 关闭页面后再次发送测试通知，确认系统通知仍能到达。
5. 打开通知，确认能够回到对应告警页面。

浏览器厂商的推送网络在服务器所在网络或设备网络不可达时，通知可能延迟或失败；前台页面中的告警记录不受影响。

## 八、常见问题

- 容器不断重启：检查 `.env` 是否存在，`APP_ENCRYPTION_KEY` 是否为 Base64 编码的 32 字节值，并查看 `docker compose logs app`。
- 页面可以打开但通知无法订阅：确认地址为有效 HTTPS、浏览器通知权限未被禁用、`PUBLIC_BASE_URL` 与访问地址一致。
- 内网 chatgpt2api 无法添加：确认确实需要访问自有内网服务后，把 `ALLOW_PRIVATE_TARGETS` 设为 `true` 并重建应用容器。
- 实时状态长时间不更新：检查反向代理是否缓冲了事件流，Nginx 可直接采用项目示例。
- 健康检查失败：先在服务器执行 `curl -fsS http://127.0.0.1:8080/healthz`，再检查容器日志和数据卷剩余空间。
