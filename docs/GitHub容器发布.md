# GitHub 容器发布

## 一、自动构建规则

仓库中的“构建容器镜像”工作流按以下规则运行：

- 每次触发先运行服务端测试、代码检查、前端测试与前端构建，全部通过后才进入镜像任务。
- 提交到主分支时，构建 AMD64、ARM64 镜像并发布到 GitHub 容器仓库。
- 推送形如 `v1.2.3` 的版本标签时，发布对应版本、主版本和次版本标签。
- 创建或更新拉取请求时，仅验证两种架构都能成功构建，不登录容器仓库，也不推送镜像。

主分支会生成 `latest`、`main` 和提交摘要标签；版本标签会生成 `1.2.3`、`1.2`、`1` 和提交摘要标签。镜像地址为：

```text
ghcr.io/账号/仓库:标签
```

## 二、首次发布

创建 GitHub 仓库并推送主分支后，在仓库的“Actions”页面查看“构建容器镜像”。工作流成功后，可在仓库右侧的“Packages”区域看到镜像。

工作流仅使用 GitHub 自动提供的令牌，发布任务拥有读取代码和写入容器包的权限，拉取请求任务只拥有读取代码的权限，不需要另外保存容器仓库密码。

若仓库或组织关闭了工作流写入包的能力，需要在仓库设置中允许 GitHub Actions 创建和写入包，然后重新运行失败的任务。

## 三、发布版本

确认主分支已通过测试后创建版本标签：

```bash
git tag v1.0.0
git push origin v1.0.0
```

标签应遵循 `v主版本.次版本.修订版本` 格式。已有标签不要重复移动到其他提交；需要修正时递增修订版本并创建新标签。

## 四、镜像可见性

GitHub 容器包的可见性由包设置单独控制。公开部署可把包设为公开；私有部署保持私有，并在服务器上使用仅具备 `read:packages` 权限的令牌登录：

```bash
printf '%s' "$GHCR_TOKEN" | docker login ghcr.io -u GitHub账号 --password-stdin
```

不要把令牌写入仓库、Compose 文件或命令历史。登录完成后，按《部署说明》设置 `POOLWATCH_IMAGE` 并拉取镜像。

## 五、回退版本

部署前记录当前镜像标签。需要回退时，把 `.env` 中的 `POOLWATCH_IMAGE` 改回上一个已验证版本，然后执行：

```bash
docker compose pull app
docker compose up -d --no-build
docker compose ps
```

镜像更新和容器重建不会自动删除业务数据卷。回退前仍应按《备份与恢复》完成数据备份。

## 六、从 GitHub 手动部署服务器

仓库提供“手动部署服务器”工作流。首次使用前，在仓库 Actions Secrets 中保存以下内容：

- `DEPLOY_HOST`：服务器地址。
- `DEPLOY_PORT`：SSH 端口。
- `DEPLOY_USER`：SSH 用户名。
- `DEPLOY_PASSWORD`：SSH 和 sudo 使用的密码。

工作流会固定校验服务器的 SSH 主机指纹。服务器重装或主机密钥变化后，应先在可信渠道核对新指纹，再修改工作流中的预期值。

镜像构建成功后，在 Actions 页面打开“手动部署服务器”，填写完整镜像地址和该镜像对应的 40 位提交编号。建议填写带摘要的不可变地址，例如：

```text
ghcr.io/账号/仓库@sha256:镜像摘要
```

部署过程会先生成 SQLite 一致性备份，再更新应用容器，并检查镜像提交编号、数据库完整性、新接口、HTTPS 健康状态和前端资源。验证失败时会自动切回原镜像配置；备份仍保留在服务器的 `/opt/poolwatch/backups`。
