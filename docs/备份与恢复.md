# 号池监控备份与恢复

## 一、必须保留的内容

完整恢复需要同时具备以下两部分：

1. `poolwatch_app-data` 数据卷的备份，其中包含管理员、渠道、加密后的凭据、历史记录、告警、推送订阅、加密后的 VAPID 私钥和设置。
2. 部署目录中的 `.env`，尤其是原始 `APP_ENCRYPTION_KEY`。

只有数据卷而没有匹配的 `APP_ENCRYPTION_KEY`，渠道凭据、TOTP 和 VAPID 推送私钥将无法解密。不要通过聊天、邮件或代码仓库传输 `.env`；应把它加密保存到与数据备份不同的位置。

使用随项目提供的 Caddy 时，`poolwatch_caddy-data` 和 `poolwatch_caddy-config` 只保存证书与 Caddy 状态。它们可以另行备份，也可以在恢复后由 Caddy 重新申请证书。

## 二、备份策略

- 至少每天备份一次，重要配置变更和版本更新前额外备份。
- 保留多个时间点，例如最近 7 份每日备份和最近 4 份每周备份。
- 至少有一份备份存放在另一台主机或受控的对象存储中。
- 定期校验 SHA-256 文件，并在隔离环境实际演练恢复。
- 备份目录仅允许管理员读取。

应用使用 SQLite WAL 模式。随附脚本会短暂停止 `app` 容器，再归档整个 `/data` 数据卷，避免数据库主文件与 WAL 文件处于不同时间点。

## 三、在 Linux 服务器备份

推荐把备份保存在项目目录之外：

```bash
sudo install -d -m 700 /srv/backups/poolwatch
cd /opt/poolwatch
sudo sh scripts/backup.sh /srv/backups/poolwatch
```

脚本完成后会自动重新启动应用，并生成两个文件：

- `poolwatch-日期-时间.tar.gz`：数据卷归档。
- 同名 `.sha256`：完整性校验值。

验证最新备份：

```bash
cd /srv/backups/poolwatch
sha256sum -c poolwatch-20260719-180000.tar.gz.sha256
```

把示例文件名替换为实际文件名。备份后还要把 `.env` 加密保存到独立位置；不要把明文 `.env` 和公开可下载的归档放在一起。

首次运行时脚本可能先下载一个轻量辅助镜像，下载发生在停止应用之前。脚本执行期间应用会短暂不可用。如果脚本失败，它仍会尝试重新启动 `app` 容器，之后应运行 `docker compose ps` 和健康检查确认状态。

## 四、Windows 开发环境临时备份

PowerShell 脚本只用于本地开发，默认把归档放到 `D:\tmp\号池监控备份`：

```powershell
powershell -ExecutionPolicy Bypass -File .\scripts\backup.ps1
```

也可以明确指定目录：

```powershell
powershell -ExecutionPolicy Bypass -File .\scripts\backup.ps1 -BackupDirectory D:\tmp\指定备份目录
```

`D:\tmp` 中的文件视为临时开发文件，不应作为正式环境唯一备份。

## 五、恢复前检查

恢复会覆盖当前数据，开始前必须完成以下事项：

1. 确认归档来源可信，并验证 SHA-256。
2. 先为当前数据再做一份备份。
3. 准备与该归档匹配的 `.env` 和 `APP_ENCRYPTION_KEY`。
4. 尽量使用生成该备份时相同的应用版本，恢复成功后再升级。
5. 通知使用者进入维护窗口，恢复过程中不要操作页面。

校验归档：

```bash
cd /srv/backups/poolwatch
sha256sum -c poolwatch-20260719-180000.tar.gz.sha256
```

## 六、恢复到原服务器

先把与备份匹配的 `.env` 放回 `/opt/poolwatch/.env` 并设置为仅管理员可读。以下命令会清空当前业务数据卷，逐项核对项目目录、容器名、数据卷名和归档文件名后再执行：

```bash
chmod 600 /opt/poolwatch/.env
```

确认无误后执行：

```bash
cd /opt/poolwatch
docker pull alpine:3.22
docker compose stop app

CONTAINER_ID="$(docker compose ps -a -q app)"
VOLUME_NAME="$(docker inspect --format '{{range .Mounts}}{{if eq .Destination "/data"}}{{.Name}}{{end}}{{end}}' "$CONTAINER_ID")"
printf '即将恢复到数据卷：%s\n' "$VOLUME_NAME"

docker run --rm \
  --volume "$VOLUME_NAME:/target" \
  --volume /srv/backups/poolwatch:/backup:ro \
  alpine:3.22 \
  sh -ec 'find /target -mindepth 1 -maxdepth 1 -exec rm -rf -- {} +; tar -xzf /backup/poolwatch-20260719-180000.tar.gz -C /target; chown -R 10001:10001 /target'

docker compose start app
docker compose ps
curl -fsS http://127.0.0.1:8080/healthz
```

如果 `VOLUME_NAME` 为空，立即停止，不要运行恢复容器。常规部署下名称应为 `poolwatch_app-data`，但命令以容器实际挂载为准。

恢复后登录并检查：

- 渠道、阈值和系统设置是否完整。
- 最新历史记录和告警是否符合备份时间。
- chatgpt2api 脱敏账号状态能否重新检测。
- 已订阅设备能否收到测试通知。

## 七、恢复到新服务器

1. 在新服务器放置与备份匹配版本的项目文件。
2. 恢复原 `.env`，并把 `PUBLIC_BASE_URL`、域名和邮箱按新环境调整；`APP_ENCRYPTION_KEY` 必须保持原值。
3. 运行 `docker compose up -d` 创建容器和空数据卷，然后立即运行 `docker compose stop app`。
4. 按“恢复到原服务器”的步骤把归档解压到新数据卷。
5. 启动应用并完成健康检查、登录、渠道检测和通知测试。
6. 域名切换完成并确认服务正常后，再下线旧服务器。

如果正式域名不变，浏览器中的推送订阅通常可以继续使用；若域名发生变化，需要在新域名下重新允许通知并订阅设备。

## 八、备份故障处理

- 提示找不到 `app` 容器：先运行 `docker compose up -d`，确认容器已创建并处于运行状态。
- 提示找不到 `/data` 数据卷：检查是否使用了项目提供的 `docker-compose.yml`，不要手工改变容器内挂载路径。
- 归档后应用没有恢复：运行 `docker compose start app`，随后检查日志和 `/healthz`。
- 校验值不匹配：不要恢复该文件，从另一份备份重试。
- 恢复后无法解密渠道凭据：确认 `.env` 中的 `APP_ENCRYPTION_KEY` 与创建备份时完全一致，不要生成新密钥替代。
